Feature Name: Password Reset
User Story: As a user, I want to reset my password using my email and full name so that I can regain access to my account if I forget my password.
Value: Prevents users from being permanently locked out of their budgeting data, while keeping the process secure by requiring identity verification.
Layers it touches: UI, Backend, and Database. 

1. PROBLEM STATEMENT
Users of the budgeting app may forget their password over time, especially
since they may not log in frequently. Without a password reset option, they
are permanently locked out of their data. This feature allows users
to securely reset their password by verifying their email address and full
name both of which must match an existing account. This protects user data
from being changed by unauthorized parties while still giving real users a
clear path back into their account.

2. USER FLOW
Step 1: User navigates to the Login page.
Step 2: User clicks "Forgot Password?" link below the login and create account buttons.
Step 3: User is taken to the Password Reset page.
Step 4: User enters their registered email address and full name.
Step 5: System checks the database for an account where both email AND
        full name match.
Step 6a: If no match is found display an error message:
         "No account found with that email and name combination."
Step 6b: If a match is found allow a new password on the same page
         asking the user to enter and confirm a new password.
Step 7: User enters a new password and confirms it.
Step 8: System validates that both fields match and password meets
        minimum requirements (at least 6 characters).
Step 9: System hashes the new password and updates the database.
Step 10: Success message is shown: "Your password has been reset. You may
          now log in."
Step 11: User is redirected to the Login page after 3 seconds.

3. TECHNICAL PLAN
Frontend Changes:
  - Add "Forgot Password?" link to the existing Login page
  - Create new page/component: ForgotPassword (passreset.html)
        -Form fields: Email, Full Name
        -On match: A second section in the form with New Password + Confirm -Password fields
        -Display inline error and success messages
        -Redirect to login on success

Backend Routes:
  - POST /api/auth/verify-identity
      Accepts: { email, fullName }
      Checks DB for matching account
      Returns: { verified: true/false, userId }

  - POST /api/auth/reset-password
      Accepts: { userId, newPassword, confirmPassword }
      Validates passwords match + length
      Hashes new password with bcrypt
      Updates users table
      Returns: { success: true/false, message }

4. API CHANGES
NEW ROUTES:

  POST /api/auth/verify-identity
  Request Body:
    {
      "email": "user@example.com",
      "fullName": "Lisa Lane"
    }
  Response (success):
    {
      "verified": true,
      "userId": 123
    }
  Response (failure):
    {
      "verified": false,
      "message": "No account found with that email and name combination."
    }

  POST /api/auth/reset-password
  Request Body:
    {
      "userId": 123,
      "newPassword": "securePass123",
      "confirmPassword": "securePass123"
    }
  Response (success):
    {
      "success": true,
      "message": "Password successfully reset."
    }
  Response (failure):
    {
      "success": false,
      "message": "Passwords do not match." | "Password must be at least 6 characters."
    }

5. DATABASE CHANGES
------------------
No new tables required.

Existing Table Modified: users

  Existing fields used (no changes needed):
    - email        VARCHAR  — used to match identity
    - full_name    VARCHAR  — used to match identity
    - password     VARCHAR  — will be updated with new hashed password

  Optional field to add (recommended):
    - password_reset_at   TIMESTAMP   NULL   DEFAULT NULL
      Purpose: logs the last time a password was reset (audit trail)

  SQL to add optional field:
    ALTER TABLE users
    ADD COLUMN password_reset_at TIMESTAMP NULL DEFAULT NULL;

6. ACCEPTANCE CRITERIA
-----------------------
[ ] User can access the Forgot Password page from the Login page
[ ] Form accepts email and full name inputs
[ ] If email + full name do NOT match any account, an error message is shown
    and no password change occurs
[ ] If email + full name DO match, user sees the new password form
[ ] New password and confirm password must match — mismatch shows error
[ ] Password must be at least 8 characters — violation shows error
[ ] On valid submission, password is updated in the database (hashed)
[ ] Success message is displayed after reset
[ ] User is redirected to Login page after success
[ ] Feature can be fully demoed without any other team member's feature
[ ] Reloading after reset requires the new password to log in (persisted)
[ ] Error handling works for all edge cases (missing fields, bad match,
    password mismatch, too short)
